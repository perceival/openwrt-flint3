# IPQ5332 on-SoC 2.4 GHz Q6/WCSS — PAS auth findings (GL.iNet Flint 3)

Status of the integrated 2.4 GHz radio (the IPQ5332 Q6/WCSS coprocessor, driven by
ath12k-AHB + a multipd remoteproc). The 5/6 GHz QCN6274 PCIe radios, LAN (RTL8372N DSA),
and 2.5 G WAN already work on mainline; this is the last piece.

## TL;DR — the `-22` PAS auth blocker is cracked (root cause + fix)

Bringing up the Q6 always failed at `qcom_scm_pas_auth_and_reset()` with **-22 (EINVAL)**.
The cause is **device-/TrustZone-specific**, not a Linux bug:

> The vendor TrustZone (QSDK QSEE) shipped on retail IPQ5332 gear implements
> `PAS_AUTH_AND_RESET` with **three** arguments — `(peripheral, debug, reset_cmd_id)` —
> whereas mainline (`qcom_q6v5_mpd` *and* `qcom,ipq5332-wcss-sec-pil`) calls it with **one**.
> The single-arg form is rejected with `-EINVAL`.

Fix = **patch `0606-remoteproc-qcom-q6-mpd-3arg-pas-auth.patch`**: adds
`qcom_scm_pas_auth_and_reset_mpd(peripheral, debug, reset_cmd_id)` and threads a per-SoC
`reset_cmd_id` (**0x18** for the IPQ5332 root PD, 0x14 for IPQ5018, from the QSDK
`q6_ipq5332_res_init`) through the driver.

**Confirmed on hardware:** the error moves from **-22 → -12 (ENOMEM)** — i.e. the TZ now
*accepts and processes* the auth and proceeds further. This matters for *any* retail IPQ5332
device: the in-flight upstream secure-PIL path uses the 1-arg auth and will `-22` on this TZ.

## Remaining blocker: `-12` (ENOMEM)

With the auth accepted, the TZ then returns **-ENOMEM** during authenticate-and-reset.
Hypotheses tested on hardware and **ruled out**:
1. allocating the bootargs/userpd-info SMEM segment (507) before auth — still -12;
2. `reset_cmd_id` value (0x14 vs 0x18) — same -12 (the arg *count* cleared -22, not the value);
3. enabling the Q6 memory/bus clocks — the QSDK doesn't enable them for the mem-protection
   path (the TZ does), so this was a wrong theory;
4. adding the four low reserved-memory regions the vendor has but our DT lacked
   (`uboot/sbl/tz/smem` filling `0x4a100000-0x4a900000`) — correct fix to keep, but still -12.

The likely-remaining lead is the upstream memory combo (`q6_fw0.mbn` merged firmware +
a 43 MB `q6_region`), but **the IPQ5332 `.mbn` is not publicly published** (absent from
quic/upstream-wifi-fw and linux-firmware). So `-ENOMEM` is currently blocked on the
upstream firmware and/or TZ-side knowledge.

## Reproduce

1. Build for `glinet_gl-be9300` (qualcommbe/ipq53xx) with the multipd patches + 0606 +
   ATH12K_AHB enabled, and the IPQ5332 ath12k firmware staged to `/lib/firmware/IPQ5332/`.
2. Enable `remoteproc@d100000` + `wifi@c000000` in the board DTS (`status = "okay"`).
3. TFTP-boot the initramfs and `dmesg | grep -aE 'ath12k|remoteproc|wcss|q6'` — you should see
   the multipd bind, `Booting fw image IPQ5332/q6_fw0.mdt`, then `can't start rproc: -12`
   (with 0606) instead of `-22` (without).
